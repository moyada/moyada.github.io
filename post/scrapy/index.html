爬虫与反爬虫

2020-03-23 2 分钟阅读时长   812 字

背景

爬虫与反爬虫是个在软件开发行业里一直存在的斗争。爬虫，一般常见的有窃取同行数据、分析行业数据、恶意攻击等，最终都是以牟利为目的。为了防止爬虫，反爬虫就得做出防护，常见的有分析请求 User-agent、加密请求数据、限制游客请求、设置用户 ip 请求频次。

然而，再完善的拦截，还是会有被突破的可能，这一切都是成本的博弈。当爬虫行为的 ROI 远远低于预期时，那么反爬虫就是成功的。

下面就来分析一下这些常见的爬虫与反爬虫手段都有哪些不同。

爬虫

API

一般情况下，爬虫都是直接调用网站内请求数据的接口，这是成本最低的方式。通过分析接口请求规则，组装成 HTTP 请求，模拟正常用户操作。

模拟浏览器

在一些网站对接口升级防护后，可能原有的方式就失效了，这个时候再尝试分析请求接口就可能徒劳无功。但是，通过模拟正常用户操作的话，那不是又能够越过防护了，这就使得模拟浏览器的爬虫开始流行起来。

这类爬虫手段可以像手工操作一样模拟，登陆、验证码、获取网页元素，为了提高效率还可以设置禁止加载图片和 css。

代理池

上述的爬虫方式都是直接使用本机来请求，那就使得所有请求都是一个请求 ip，很有可能进入风控。这时可以通过设置请求代理来切换身份，一般程序都会维护一个 ip池，让每次请求都从这个代理 ip 池中获取代理 ip 再操作。

反爬虫

Header 拦截

在网站的请求用户可预知的情况下，可以设置一些 Header、User-agent、Cookie 来简单拦截非法请求。

验证码

为了进一步防止爬虫行为，通常会对一些关键接口增加验证码校验，比如字符验证码、问答验证码、滑动验证码。

数据加密

在一些电商网站的交易流程里，会对接口的请求数据进行加密，使得分析接口爬虫变得几乎不可能。在一些数据重要性高的行业中，还会对页面数据进行混淆，利用直接返回 html 代码或使用 css 渲染数据。

频次拦截

当正常流程里的防护措施都失效时，就需要系统对请求数据进行分析，识别爬虫行为，可以使用滑动窗口对归类请求进行计数，对爬虫用户进行拦截配置。
